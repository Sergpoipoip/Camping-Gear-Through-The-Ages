Abstract

The Evolution of Writing is a digital companion to a temporary exhibition that traces 20 000 years of human communication, from Ice‑Age tally sticks to the Unicode Standard. The web application invites visitors to explore sixteen artefacts that represent turning‑points in how societies record, preserve and share knowledge.

Combining a responsive interface, narrative modes and multimodal assets (timeline, maps, object cards), the project bridges the on‑site visit with remote discovery and classroom use.

1. Introduction

1.1 Overview

The site is designed to provide the visitor with a coherent and flexible exploration of the history of writing, whether their motivation is curiosity, research, or pedagogy. The website is organised around three complementary access paths — Historical Themes, Object Types and Periods. Each path reshuffles the same data set, encouraging users to look at the artefacts through different interpretive lenses.

The entirety of the exhibition content is managed in a structured JSON format, with all metadata, descriptive texts, and references defined and versioned for stability and maintainability. Thisrchitecture underpins the flexibi ality of the user experience and supports future extension or integration with other digital humanities initiatives.

1.2 Objectives

  • Engagement. Offer an immersive, story‑driven experience ahead of — or in parallel with — the physical display.
  • Education. Provide layered descriptions (short, longer, full) so that pupils, casual visitors and scholars all find a suitable depth of information.
  • Findability. Allow users to reach any artefact in ≤3 clicks via narrative buttons, the filters on the Items page, the timeline or the interactive maps.
  • Accessibility. Bootstrap 5 ensures semantic HTML, keyboard navigation, ARIA labels and responsive layouts out of the box.

1.3 Requirements

The full data set is stored in a single JSON file. items.json contains all cataloguing metadata plus the multi‑level narratives for each artefact.

2.3 Narratives

Our three narrative families give visitors agency in shaping their own learning journey. Rather than prescribing a single, linear storyline, the interface offers filters-as-pathways. Each pathway foregrounds a different aspect of the same artefacts, enabling serendipitous discovery and reinforcing key learning outcomes.

  • Historical Themes – Focuses on the social function of writing (e.g. Commerce & Administration, Religion & Myth). This thread helps visitors connect the objects with everyday human needs: trade, governance, spirituality.
  • Object Types – Groups artefacts by their material form (tablet, manuscript, device). By zooming in on changing substrates and tools, visitors grasp how technological constraints shaped scripts and layouts.
  • Periods – A chronological overview (Prehistoric → Modern) aids time‑travel across millennia. Juxtaposing artefacts from the same era reveals cultural exchange and parallel innovations across regions.

3. Development

3.1 Information Architecture

Information on the site is organised according to the principle of a single authoritative data source. The file items.json resides at the project root and is fetched independently by each core view — the item gallery, the timeline, and the two maps — when that view is first loaded. On every request the raw JSON is parsed into plain JavaScript objects whose scope is limited to the page that triggered the fetch.

Inside the file, all records are gathered under the top‑level key items, itself an array. Each array element is one artefact and follows a predictable internal grammar. A numeric @sort field determines chronological ordering across the site; shortName supplies card headings; image stores the path to the default image of an artefact; periodTag maps directly to the period filter. The nested info object holds authoritative catalogue metadata such as Object type, Materials, Housed in and others.

Interpretive texts are separated by granularity. shortInfo is a tweet‑length hook displayed on gallery cards; longerInfo is an array that can hold multiple paragraphs for modal pop‑ups; fullInfo is a path to an external HTML essay which is lazy‑loaded only when the user explicitly asks for the full story. This lazy approach keeps the initial payload light, while still offering limitless depth.

The same JSON powers the timeline through the @sort field and feeds the Leaflet maps via geocoded location fields generated during the build process.

3.2 User Experience (UX)

The site’s navigation is built around the principle of letting visitors approach the exhibition from multiple angles. At its centre stands the Items page, which houses every object’s catalogue entry. Here each artefact appears with a data table, an image, and a description that the visitor can expand or collapse at will; quick-jump buttons toggle among narratives, while filter controls beneath the main menu allow combinations of themes, periods, and types.

Branching out from this hub are four specialised views. The Narratives page offers concise explanations of each storyline and invites users to start exploring by simply selecting the strand that interests them. The Timeline arranges all items chronologically, giving a clear diachronic perspective; by clicking on the artefact image the user gets redirected to the Items page where a card with information about this artefact pops up.

There are also two maps: the Exhibition one and Geographical. The Exhibition Map mirrors the physical gallery layout, providing direct links to the artefacts descriptions. The Geographical Map page lets visitors follow each artefact’s real-world journey. When it loads, a Leaflet map centres on the globe and drops two colour-coded markers for every artefact — orange for the place where it was found and blue for the institution that now houses it. Markers are clustered automatically so the view stays clear at low zoom levels, and a popup on each pin links straight to that object’s full record on the Items page. A sidebar on the right offers drop-down filters for theme, typology and historical period; changing any filter instantly hides or reveals the corresponding markers, mirroring the logic used on the Items grid.

3.3 Technical Implementation

The whole web app is driven by a single authoritative data source, items.json, plus four JavaScript modules that transform that data into the various interactive views.

Loading the data. Every view that needs artefact information begins with the same promise chain: fetch('items.json') retrieves the file, response.json() parses it, and the returned data.items array is either cloned or filtered to fit the view’s purpose.

Once the array is in memory, each page transforms it in a different way that matches the questions that page tries to answer.

• Catalogue (items.html + js/items.js)

The helper function buildFilters() scans the entire items array and extracts every distinct value it finds in the Theme, Typology, and Period fields, using those sets to populate the corresponding drop-down menus. Whenever the visitor changes one of those selectors, the page fires a live-filtering cycle: the renderGrid() function reruns, discarding all records whose metadata no longer satisfies the trio of active facets and rebuilding the grid so that only matching artefacts remain on screen.

For each artefact that survives this filtering pass, the script constructs a Bootstrap card on the fly, injecting the item’s teaser image, its shortName, and a two-line snippet taken from shortInfo. Pressing a card’s Details button triggers showModal(). The modal routine receives the artefact’s internal @sort identifier, looks the object up in memory, and assembles an enlarged hero image, a tidy metadata table derived from the info block, and — when available — long and full narrative sections.

• Timeline (timeline.html + js/timeline.js)

The timeline script first clones the artefact array and sorts it numerically by @sort, guaranteeing that objects are laid out in genuine chronological order. As it walks through this ordered list it watches for changes in periodTag; whenever a new period is encountered it inserts a header tile, breaking the strip into clearly labelled historical eras.
To keep the pictures from piling up, each successive artefact tile is then placed alternately above and below the central axis — an effect achieved with a simple Boolean toggle that flips on every iteration.
Hovering tile reveals an overlay that repeats the shortName and provides a “See more” link back to the catalogue modal, allowing the user to drill straight into full object details.

• Geographical map (map.html + js/map.js)

When the map view loads it first “harvests” the metadata it needs for later filtering. For every artefact the script splits the semicolon-separated Subjects string, reads the Object type, and notes the single periodTag; each of those three facets is stored as a Set directly on the Leaflet marker so they can be compared in constant time during interactive filtering.

Next the code resolves geography. The free-text place names found in Original Location and Housed in are fuzzily matched against a hand-curated PLACES dictionary, yielding accurate latitude-and-longitude pairs (and, where relevant, a museum URL).
With coordinates in hand the script creates up to two pins for every artefact — an orange one that marks its provenance and a blue one that shows its present repository. Each pin opens a Bootstrap-styled popup containing a link such as items.html?sid=…, allowing the visitor to jump straight to the artefact’s catalogue modal without losing context.

All pins are added once to a single MarkerClusterGroup; afterwards the three facet dropdowns (Theme, Typology, Period) simply add or remove markers from that cluster layer in real time, applying exactly the same logic as the catalogue grid but entirely on the client side. This approach keeps the map responsive, even as hundreds of artefacts are filtered in milliseconds.

4. Use Cases

The interface delivers progressive immersion, surfacing different narrative lengths to suit three personas:

  1. Marco, 15, secondary‑school student.
    • Goal: Prepare a 5‑minute class presentation on “the origins of writing”.
    • Journey: Uses the Timeline to grasp chronology → filters by Prehistoric period → reads short descriptions.
  2. Alice, 34, middle-school history teacher
    • Goal: Craft an engaging 40-minute lesson on the Rosetta Stone for her Year 8 class.
    • Journey: Opens the Items → filters by “Governance & Law” theme → selects the Rosetta Stone card → expands the longer description accordion to grab the discovery story and key dates → projects the modal (image + long text) on the smartboard during class → clicks Full story and posts that deep-dive link to the class portal for homework reading.
  3. Dr. Ibrahim, 55, epigraphy scholar.
    • Goal: Investigate comparative writing technology.
    • Journey: Direct deep link (shared by curator) opens Unicode Standard 1.0 modal in full description mode → exports catalogue data via the /api/items endpoint → toggles map current location layer to confirm object custodianship.

These flows demonstrate how the same data is repackaged for snackable, guided and scholarly consumption.

5. Design Specifications

The visual language of Evolution of Writing is built around the metaphor of a scholar leafing through illuminated manuscripts. Parchment hues, ink-dark text, and quiet micro-animations evoke the tactility of historical documents while Bootstrap’s mobile-first grid and Leaflet maps keep the experience unapologetically contemporary.

5.1 Layout

5.1.1 Homepage

The landing page is divided into three vertically stacked “folds”. The first fold is a full-viewport hero carousel: .carousel-item { height:100vh; } makes each slide occupy the entire browser window, ensuring that the opening imagery commands attention before the visitor scrolls. We chose 100 vh rather than a fixed pixel height so that ultrawide monitors and short laptop screens enjoy the same immersive ratio.

The second fold introduces the Explore grid. A CSS Grid layout renders four equally wide columns on desktop (repeat(4, minmax(120px, 1fr))) but collapses gracefully to two columns below 576 px, maintaining finger-friendly hit areas on phones. Because the grid is fed by JS from items.json, the site can grow new themes without touching HTML.

The third fold, Find Us, uses a 60 / 40 split (#find-us { grid-template-columns: 3fr 2fr }) to pair a Leaflet map with a fact sheet. We intentionally inverted the common “map-sidebar” pattern — placing the map on the left — because Western reading starts at the left margin; users therefore register the geospatial context before scanning the address, a cognitive order that matches how one would approach a real museum catalogue.

5.1.2 Narratives

The narratives index functions as a tiled magazine cover. Bootstrap’s utility class .row-cols-lg-5 outputs five cards across on large screens, four on medium, down to two on extra-small — giving us perfectly fluid columns without custom breakpoints. Each card is an aspect-ratio box (aspect-ratio:3/2) so images never warp, and two ::before/::after pseudo-elements draw folded-corner ornaments, reinforcing the “paper page” metaphor.

5.1.3 Timeline

The timeline view is a horizontal strip (display:flex; gap:3.5rem) inside an overflow container that hides the native scrollbar for visual purity (.timeline::-webkit-scrollbar { display:none }).
To prevent images from colliding with the central line we alternate tile positions using utility classes .tl-item.above and .below, which translate the element −70 px and +90 px respectively.

5.1.4 Map

Both Geographical Map and Exhibition Layout pages are built on a flex container named .page-body: above 992 px the direction is row (map | sidebar), below 992 px it flips to column so that filters never squeeze between viewport edges. We limited the sidebar to max-width:33.333% to prevent text columns from exceeding 70 characters, in line with legibility recommendations by the Information Architecture Institute.
The map itself inherits the parchment background through CSS override (.leaflet-container { background:transparent !important; }), ensuring tiles appear to rest on the same material as the rest of the page. All markers share a single Leaflet MarkerClusterGroup.

5.2 Colour Palette

Colours are declared once in :root so every component, regardless of page, pulls from the same ink-and-parchment scheme. The background tone #f6f1e0 was sampled from aged vellum and lightened by 4 % to hit a 14.8:1 contrast ratio against the charcoal text #2e2e2e, comfortably surpassing WCAG AA for body copy at 16 px.

  • Sepia Ink — --color2
    Used for headings and link default states. Sepia reads as “ink that has oxidised” and therefore signals hierarchy without resorting to pure black, keeping large headings visually lighter.
  • Rusty Red Wax — --color5
    Buttons, hover states and progress indicators adopt a translucent #cc3112; the hex value was tuned to a L* of 39 in the CIELAB colour space so white text overlays still meet 4.5:1 contrast. The transparency (0x9d alpha) allows the parchment grain to remain subtly visible, echoing the semi-opaque nature of historical sealing wax.
  • Muted Navy — --color3
    A cool counter-accent (#37475a) balances the warm palette. We deploy it sparingly: progress bars, secondary buttons and the Explore panel backdrop. The aim is to avoid “visual fatigue”.

5.3 Typography

Typography delivers half the historical mood. We pair two Google Fonts, loaded once in the document head to avoid FOIT flashes.

  • Playfair Display (Headers)
    A transitional serif reminiscent of engraved title pages, used for all headings and narrative-card labels via --font-family-header. Its high contrast strokes give engraved-ink elegance, aligning with the site’s scholarly tone.
  • Crimson Text (Body)
    Selected for running text because its loose x-height and moderate stroke contrast mirror Garamond yet ship lighter, improving LCP on slower connections. Every paragraph, card snippet and modal narrative inherits font-family:var(--font-family-text), guaranteeing a uniform reading texture across pages, including pop-ups.

The navigation bar adopts geometric Montserrat and buttons fall back to pragmatic Tahoma; this deliberate divergence helps users instinctively separate actionable controls from readable content. Typography sizes obey a 1.250 Major Third scale: body 16 px, h6 20 px, h1 ≈40 px. The scale was chosen because a 1.25 ratio approximates the inverse golden ratio (φ-1) championed in classical book design.

Collectively, these layout, colour and typography choices fuse modern responsiveness with period-appropriate ambience, underscoring our central design aim: to let users feel as though they are browsing a digital codex, not merely a web page.
